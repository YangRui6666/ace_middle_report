## ACE 电控软件编程规范 - V1.5
这是一份东莞理工 ACE 实验室电控组的嵌入式开发指引。
作为 ACE 团队的一员，不同的开发任务会由不同人员负责，但如果没有统一的代码规范，每个人的代码风格各异，会导致许多本可通用的功能需要重复编写，造成事倍功半、效率低下的问题。因此，结合学长的建议、RT-Thread 编程风格、大疆开源代码风格及个人经验，汇总整理了这份代码规范参考。
请每一位东莞理工 ACE 实验室电控组的嵌入式开发者遵守该编程风格。
同时，新人也可通过这份文档了解代码内部的相关约定，从而更快融入各类开发实战。

### 1. 目录名称
无特殊需求时，目录名称请使用全小写形式。
目录名称应能体现其包含内容的含义，例如各芯片移植相关目录以芯片名称或芯片类别命名，components 目录下的子目录需能反映对应组件的功能。

### 2. 文件名称
无特殊需求时（若引用其他来源的文件，可保留其原有名称），文件名称请使用全小写形式。
为避免文件名重复，应尽量避免使用通用性强、使用频率高的名称。

### 3. 结构体定义
结构体名称采用小写英文命名，单词之间用下划线“_”连接，示例如下：
```c
typedef struct
{
    const motor_measure_t *chassis_motor_measure;
    fp32 accel;
    fp32 speed;
    fp32 speed_set;
    int16_t give_current;
} chassis_motor_t;
```
其中，左大括号“{”需独立占用一行，右大括号“}”后空一格，紧跟结构体名称并加上“_t”作为类型名。
结构体等类型的定义示例：
```c
// 底盘控制数据
static chassis_control_t chassis_control;
```
为实现各文件间对象的访问，采用返回结构体地址的方式，在其他文件中接收，示例如下：
```c
/**
 * @brief 返回遥控器控制变量，通过指针传递方式传递信息
 * @param[in] none
 * @retval 返回遥控器控制变量 &rc_ctrl
 * @attention
 */
const rc_ctrl_t *get_remote_control_point(void)
{
    return &rc_ctrl;
}
```

### 4. 函数名称、声明
函数名称采用小写英文命名，单词之间用下划线“_”连接。
提供给上层应用使用的 API 接口，必须在对应的头文件中声明。
若函数入口参数为空，必须使用 void 作为入口参数，示例如下：
```c
/**
 * @brief 返回遥控器控制变量，通过指针传递方式传递信息
 * @param[in] none
 * @retval 返回遥控器控制变量 &rc_ctrl
 * @attention
 */
const rc_ctrl_t *get_remote_control_point(void)
{
    return &rc_ctrl;
}
```

### 5. 注释编写
注释用于解释复杂算法的工作原理。
对语句的注释仅可写在其上方或右方，其他位置均为不规范写法。
若函数逻辑复杂或具有特殊性，可在函数上方添加详细解释。

### 6. 缩进及分行
缩进采用 4 个空格实现。
无特殊情况时，左大括号“{”后需分行，下一行开始采用缩进格式，示例如下：
```c
if (condition)
{
    /* others */
}
```
唯一例外是 switch 语句，case 语句需与 switch 对齐，后续语句采用缩进格式，示例如下：
```c
switch (value)
{
case value1:
    break;
case value2:
    break;
}
```
分行方面，无特殊考虑时，代码中不得连续使用两个及以上空行。

### 7. 大括号与空格
从代码可读性角度，建议每个大括号单独占用一行，而非紧跟在语句之后，示例如下：
```c
if (condition)
{
    /* others */
}
```
匹配的大括号单独成行，可使代码层次更清晰，避免混淆。
空格使用规范：
- 非函数调用的括号（如 if、for、while、switch 语句后的括号）前需留一个空格，示例如下：
```c
if (x <= y)
{
    /* others */
}
for (index = 0; index < MAX_NUMBER; index++)
{
    /* others */
}
```
- 运算表达式中，运算符与操作数之间留一个空格。
- 括号内的表达式两侧不得留空格，错误示例如下：
```c
if ( x <= y )  // 不允许在括号内两侧添加空格
{
    /* others */
}
```

### 8. 函数（模块化）
编程时，函数应尽量精简，仅实现相对独立的简单功能。
函数实现不宜过长，若函数篇幅过大，需思考如何修改（或拆分）以使其更精简、易懂，且函数层级不宜过多。

### 9. Doxygen 制作程序文档
| 命令 | 字段名 | 语法 |
| --- | --- | --- |
| @file | 文件名 | file [<name>] |
| @brief | 简介 | brief {简要描述} |
| @author | 作者 | author {作者列表} |
| @mainpage | 主页信息 | mainpage [(标题)] |
| @date | 年-月-日 | date {日期描述} |
| @version | 版本号 | version {版本号} |
| @copyright | 版权 | copyright {版权描述} |
| @param | 参数 | param [(方向)] <参数名> {参数描述} |
| @return | 返回 | return {返回值描述} |
| @retval | 返回值 | retval {返回值详情描述} |
| @bug | 漏洞 | bug {漏洞描述} |
| @details | 细节 | details {详细描述} |
| @pre | 前提条件 | pre {前提条件描述} |
| @see | 参考 | see {参考内容} |
| @link | 链接（与 @see 类似，用法：{@link www.google.com}） | link <链接对象> |
| @throw | 异常描述 | throw <异常对象> {异常描述} |
| @todo | 待处理 | todo {待完成事项描述} |
| @warning | 警告信息 | warning {警告信息} |
| @deprecated | 弃用说明（可描述替代方案、预期使用期限等） | deprecated {弃用描述} |
| @example | 示例 | example {示例描述} |

注：实际编写注释时，无需包含所有参数，根据需求合理选用即可。
